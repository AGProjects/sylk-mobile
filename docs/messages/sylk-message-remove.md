# `application/sylk-message-remove`

"Delete a single message everywhere." Sent when the user deletes a message
on one device, so the server and the user's other devices (and, if the user
chose remote deletion, the original peer) drop their copies too.

- **Direction**: both — generated by the deleting device, fanned out to all
  parties that should drop the row.
- **Encrypted**: no — the body is a tiny control struct, not user content.
- **Persistence**: not retained as a chat row. `journalReplay` and the
  on-load chat restore both delete any straggling rows of this type to
  keep them out of the rendered timeline.

## Body shape

| Field     | Type   | Notes |
| --------- | ------ | ----- |
| `id`      | string | The `msg_id` of the message to delete. |
| `contact` | string | The peer URI the original message was exchanged with. The receiver uses this to locate the right conversation. |

## Example payload

Wire envelope:

```json
{
  "id": "7a8b9c0d-1e2f-3a4b-5c6d-7e8f9a0b1c2d",
  "sender":   { "uri": "alice@sip2sip.info" },
  "receiver": "alice@sip2sip.info",
  "timestamp": "2026-05-07T09:14:00.000Z",
  "contentType": "application/sylk-message-remove",
  "content": {
    "id":      "f3d7a14e-8b62-4f70-9d0c-2e5e7c2c9a44",
    "contact": "bob@sip2sip.info"
  }
}
```

(Receivers read `message.content.contact` to resolve the conversation; see
`app.js`: `if (message.contentType === 'application/sylk-message-remove')
{ uri = message.content.contact; }`.)

## Notes

- Generated indirectly via `addJournal(msg_id, 'removeMessage', {uri: uri})`
  in `deleteMessage()`. The journal entry is replayed by
  `replayJournal()`, which calls
  `account.removeMessage({id, receiver: data.uri}, …)` — that's the call
  that produces the wire envelope above.
- For meeting-handshake bubbles, the deletion side adds the *paired*
  session id to the journal too, so both sides of a request/accept pair
  are wiped together. See the `_session.requesterOriginId /
  accepterOriginId` block in `deleteMessage`.
